Authentication

Every endpoint under /api/* and POST /soap requires authentication. Two schemes are supported and can be used interchangeably.

Basic Authentication

Encode client_id:secret as Base-64 and send it in every request.

Authorization: Basic dGVzdC1jbGllbnQ6dGVzdC1zZWNyZXQ=

OAuth 2.0 — Client Credentials

Exchange your credentials for a short-lived bearer token, then send that token on every request.

Authorization: Bearer <access_token>
Obtaining a Bearer Token
POST /oauth/token

Send credentials as application/x-www-form-urlencoded or via HTTP Basic Auth header.

Request body (form-encoded)
FieldTypeRequiredDescription
grant_typestringrequiredMust be client_credentials
client_idstringoptional*Client ID from config. Can be sent in Basic Auth header instead.
client_secretstringoptional*Secret from config. Can be sent in Basic Auth header instead.
Response — 200 OK
{
  "access_token": "a1b2c3d4e5f6a1b2c3d4e5f6a1b2c3d4",
  "token_type":   "Bearer",
  "expires_in":   3600
}

Response Formats & Headers

All REST endpoints return JSON by default. Switch to XML in two ways:

  • Send Accept: application/xml (or Accept: text/xml) in the request header.
  • Append ?format=xml to the URL — this takes priority over the header.

For requests that carry a body, set Content-Type: application/json for JSON or Content-Type: application/xml for XML. All body examples below are shown as JSON.

The SOAP endpoint always uses XML regardless of these settings.

Pagination

All list endpoints are paginated. Use the query parameters below to navigate pages. Defaults and ceilings are configured separately for users and persons (contracts share the persons settings).

Query paramDefaultDescription
page11-based page number
page_size20Items per page. Capped at the configured maximum (default 100).

Every list response includes a pagination object alongside the data:

{
  "users": [ /* items for this page */ ],
  "pagination": {
    "page":        2,
    "page_size":   20,
    "total":       57,
    "total_pages": 3,
    "next_page":   "/api/users?page=3&page_size=20"  // empty string on last page
  }
}
Config keys: pagination.users.default_page_size / pagination.users.max_page_size and pagination.persons.default_page_size / pagination.persons.max_page_size. Persons settings also apply to all contract list endpoints.

Rate Limiting

Token-bucket rate limiting can be enabled independently for users and persons (including contracts). When disabled (default) all requests pass through freely.

Config keyTypeDescription
rate_limit.users.requests_per_10secondsfloatRequests allowed per 10-second window. 0 or negative = disabled. Divided by 10 internally to produce the per-second token-bucket rate.
rate_limit.users.burstintMaximum instantaneous requests (token-bucket burst).
rate_limit.persons.requests_per_10secondsfloatSame for persons & contracts endpoints.
rate_limit.persons.burstintBurst for persons & contracts.

When the limit is exceeded the server responds with 429 Too Many Requests and a Retry-After: 1 header.

HTTP/1.1 429 Too Many Requests
Retry-After: 1

{ "message": "rate limit exceeded — try again later" }

SOAP

The SOAP endpoint accepts standard SOAP 1.1 envelopes over HTTP POST. The same authentication rules apply.

Endpoint
POST /soap
WSDL
GET /soap
Namespace
http://testapi.local/
Content-Type
text/xml; charset=utf-8
Operation dispatch: the server reads the root element name inside <soap:Body> (e.g. <tns:GetUser>) to determine the operation. Alternatively, set the SOAPAction header to "http://testapi.local/OperationName".
Envelope skeleton
<?xml version="1.0" encoding="utf-8"?>
<soap:Envelope xmlns:soap="http://schemas.xmlsoap.org/soap/envelope/"
               xmlns:tns="http://testapi.local/">
  <soap:Body>
    <tns:OperationName>
      <!-- operation elements -->
    </tns:OperationName>
  </soap:Body>
</soap:Envelope>

OpenID Connect SSO

When oidc.discovery_url is set in config.json the server registers five browser-facing routes that implement the Authorization Code Flow. The feature is completely opt-in — leave discovery_url empty and the routes do not exist.

When is it useful? Use OIDC SSO to log in with your corporate identity provider (Hello ID, Azure AD, Okta, Google, …), inspect the claims it sends, verify your group-membership attribute mapping, and obtain an API Bearer token that carries your IdP permissions.
Configuration
Config keyTypeDescription
oidc.discovery_urlstringIdP base URL. The server appends /.well-known/openid-configuration automatically.
Examples: https://accounts.google.com · https://login.microsoftonline.com/{tenant}/v2.0
oidc.client_idstringApp / client ID registered at the IdP.
oidc.client_secretstringApp secret registered at the IdP.
oidc.redirect_urlstringMust match the callback URL registered at the IdP. Example: http://localhost:8080/oidc/callback
oidc.scopesstring[]Scopes to request. Defaults to ["openid","profile","email"]. Add provider-specific scopes to receive group claims (e.g. "groups" for Okta/Azure).
oidc.permissions_claimstringName of the ID-token claim that carries group memberships / roles. The value may be a JSON array of strings or a comma-separated string.
Common values: memberOf (Hello ID / Active Directory) · groups (Okta, Azure AD) · roles
Hello ID — group membership attribute

In Hello ID, enable Send Group Membership Attribute on the application and set the attribute name to match your permissions_claim value (e.g. memberOf). The server will read that claim from the ID token and display the memberships on the /oidc/me page.

GET /oidc/login

Redirects the browser to the IdP's authorization endpoint to start the SSO flow. No parameters required.

Response
StatusCondition
302Redirect to IdP authorization URL. A short-lived oidc_state cookie is set for CSRF protection.
GET /oidc/callback

IdP redirect target. Exchanges the authorization code for tokens, verifies the ID token, extracts the permissions claim, and creates a server-side session.

Query parameters (set by the IdP)
ParamDescription
codeAuthorization code from the IdP
stateAnti-CSRF nonce — must match the oidc_state cookie
Response
StatusCondition
302Redirect to /oidc/me on success. Sets an oidc_session cookie (HttpOnly).
400State mismatch (CSRF) or expired state
500Token exchange or ID-token verification failure
Register this URL as the allowed redirect / callback URI at your IdP. The exact value must match oidc.redirect_url in config.
GET /oidc/me

HTML profile page for the currently logged-in SSO user. Requires an active oidc_session cookie; redirects to /oidc/login if not present.

Page sections
SectionDescription
ProfileAvatar (if picture claim present), display name, and email
PermissionsAll values from the configured permissions_claim shown as tags. If the claim is absent a hint is displayed explaining how to configure the IdP attribute mapping.
ID token claimsFull claim set from the ID token as a key/value table — useful for debugging attribute mapping
API Bearer tokenButton to call /oidc/token and display the resulting token alongside the effective permissions
GET /oidc/token

Issues a standard API Bearer token for the logged-in OIDC user. The token is stored in the same token store used by POST /oauth/token and is accepted by Authorization: Bearer … on all /api/* endpoints. The response additionally carries the permissions list extracted from the IdP claim.

Response — 200 OK
{
  "access_token": "a1b2c3d4e5f6a1b2c3d4e5f6a1b2c3d4",
  "token_type":   "Bearer",
  "expires_in":   3600,
  "permissions": ["CN=HR-Admins,OU=Groups,DC=corp,DC=local", "CN=IT-Staff,OU=Groups,DC=corp,DC=local"]
}
Errors
StatusCondition
401No active OIDC session — visit /oidc/login first
GET /oidc/logout

Clears the server-side session and the oidc_session cookie, then redirects to /. Does not perform IdP-side logout (RP-initiated logout is not implemented).

Response
